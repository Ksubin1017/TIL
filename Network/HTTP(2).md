# HTTP의 특징

## 요청-응답 기반 프로토콜
HTTP는 기본적으로 요청 메시지를 보내는 클라이언트와 이에 대한 응답 메시지를 보내느 서버가 서로 HTTP 요청 메시지와 HTTP 응답 메시지를 주고받는 구조로 작동  
같은 HTTP 메시지라도 요청과 응답 메시지의 형태는 다르다.

## 미디어 독립적 프로토콜
애플리케이션의 다양한 데이터를 네트워크를 통해 송수신한다는 목적에 따라 HTTP 메시지를 통해 다양한 종류의 자원을 주고받을 수 있다.  
-> 자원의 특성과는 무관하며 자원을 주고받는 수단(인터페이스)의 역할만 수행

## Stateless 프로토콜
상태를 유지하지 않는 프로토콜  
서버는 HTTP 요청을 보낸 클라이언트 관련 상태를 기억하지 않는다.  
-> 클라이언트의 모든 HTTP 요청은 기본적으로 독립적인 요청으로 간주

HTTP 서버는 수많은 클라이언트와 동시에 상호작용하는데, 이러한 환경에서 모든 클라이언트 상태 정보를 유지하는 것은 서버에 큰 부담이 된다.  
또한 서버는 여러 대로 구성될 수 있는데 모든 서버가 모든 클라이언트의 상태 정보를 공유해야하고 이는 번거롭고 복잡하다.

HTTP 서버의 설계 목표는 확장성과 견고성으로 서버가 모든 요청을 독립적인 요청으로 처리하면 특정 클라이언트가 특정 서버에 종속되지 않는다.  
서버를 필요시 쉽게 서버를 추가할 수 있는 확장성과 장애 발생 시 다른 서버로 대체할 수 있는 견고성이 확보된다.

## 지속 연결 프로토콜
HTTP 1.1 이상부터 지속 연결(keep-alive)을 제공  
하나의 TCP 연결 상에서 여러 개의 요청-응답을 주고받을 수 있고 이를 통해 비지속 연결보다 빠른 속도로 HTTP 요청과 응답을 처리  
-> 매 요청마다 TCP 연결 수립-종료 과정을 거치지 않아도 되기 때문이다.

# HTTP 메시지 구조
시작 라인, 필드 라인, 메시지 본문으로 구성  
필드 라인은 여러 개가 존재할 수 있고 메시지 본문은 없을 수도 있다.  
메시지 본문에 HTTP를 통해 주고받는 자원이 명시된다.

HTTP는 요청-응답 기반 프로토콜이므로 HTTP 요청 메시지와 HTTP 응답 메시지가 있다.  
시작 라인으로 구분되며 시작 라인은 HTTP가 요청 메시지일 경우 요청 라인, 응답 메시지일 경우 상태 라인이 된다.

## 요청라인 구성
메서드(공백)요청 대상(공백)HTTP 버전(줄바꿈)

## 상태라인
HTTP 버전(공백)상태 코드(공백)이유 구문(줄바꿈)

## 필드 라인
필드 라인에는 HTTP 메시지 전송과 관련한 부가 정보이자 제어정보인 HTTP 헤더가 명시  
일반적으로 하나의 HTTP 메시지에는 여러 헤더가 포함되어 있고 콜론(:)을 기준으로 헤더 이름과 그에 대응하는 헤더 값으로 구성된다.
