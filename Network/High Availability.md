# 가용성(Availability)

주어진 **특정 기능을 실제로 수행할 수 있는 시간의 비율**을 의미

- **업타임(Uptime)**: 정상적으로 서비스가 제공되는 시간  
- **다운타임(Downtime)**: 장애, 트래픽 과부하, 보안 공격, 자연재해 등의 이유로 서비스가 불가능한 시간

**가용성 공식**  
가용성 = 업타임 / (업타임 + 다운타임)

**고가용성(High Availability)**  
위 수식의 값을 최대한 높이는 성질을 의미  
하지만 모든 다운타임의 원인을 원천 차단하는 것은 현실적으로 불가능하므로,  
핵심은 **문제가 발생하더라도 기능을 계속 유지하도록 설계(결함 감내)**하는 것이다.

# 로드 밸런싱(Load Balancing)

트래픽을 여러 서버로 고르게 분배하는 기술
로드 밸런서는 *다중화된 서버와 클라이언트 사이*에 위치하며 요청을 균등하게 나눈다.

- L4, L7 로드 밸런서  
- 소프트웨어 예: **HAProxy**, **Envoy**, **Nginx** 등

## 로드 밸런싱 알고리즘

### 라운드 로빈(Round Robin)
서버에 순서대로 부하를 전달하는 방식
> 서버 성능이 모두 동일할 때 효과적이다.

### 최소 연결(Least Connections)
현재 연결 수가 가장 적은 서버에 우선적으로 요청을 전달하는 방식


서버마다 처리 능력이 다를 경우 가중치를 부여하여 트래픽 분배를 조정할 수 있다.

# 스케일 업(Scale Up)과 스케일 아웃(Scale Out)

## 스케일 업(수직적 확장)

서버, 디스크, CPU, 네트워크 장비 등 **개별 장비의 사양을 높이는 방식**.

**장점**  
- 설치 및 구성 단순  
- 기존 서버를 더 좋은 장비로 교체하면 됨

**단점**  
- 유연성 부족  
- 장비 성능의 한계 → 더 비싼 장비로 교체하는 수밖에 없음

---

## 스케일 아웃(수평적 확장)

서버나 장비의 **개수를 늘리는 방식**.

**장점**
- 확장 및 축소가 유연함 → 트래픽에 따라 탄력적으로 조절 가능  
- 장비가 고장나도 다른 장비가 대신 처리 → **결함 감내성(Fault Tolerance) 높음**

**단점**
- 동일 데이터를 여러 서버가 보유해야 할 경우 데이터 일관성 문제(DRBD, Replication 등)  
- 관리 복잡도 증가
