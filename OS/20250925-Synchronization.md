# 임계 구역과 동기화

## 임계 구역(Critical Section)과 레이스 컨디션(Race Condition)

프로세스 A가 공유 메모리 공간에 데이터를 쓰고, 프로세스 B가 해당 메모리 공간을 읽을 때 두 프로세스는 공유 메모리 공간을 공유하고 있다.  
스레드 A와 스레드 B가 각각의 프로세스가 할당받은 파일을 수정하는 경우 두 스레드는 하나의 파일 자원을 공유하고 있게 된다.  
이러한 파일, 메모리, 전역 변수, 입출력 장치 등을 **공유 자원**이라고 한다.  

### 문제 상황 예시
text.txt 파일에 `Hello`라는 글자가 있다고 가정하자.

1. 스레드 A가 text.txt 파일의 내용을 `Hi`로 변경 (저장 전)  
2. 스레드 B가 text.txt 파일의 내용을 `Bye`로 변경 (저장 전)  
3. 스레드 A가 파일 저장  
4. 스레드 B가 파일 저장  

➡ 최종적으로 `Bye`만 반영되고 `Hi`는 손실된다.  

즉, **공유 자원에 동시에 접근할 때 문제가 발생할 수 있는 코드 영역을 임계 구역**이라 하고,  
**두 개 이상의 프로세스/스레드가 임계 구역을 동시에 실행하여 문제가 발생하는 상황을 레이스 컨디션**이라 한다.  

레이스 컨디션은 자원의 **일관성(consistency)** 을 손상시키므로, 임계 구역에 접근할 때는 **하나의 작업이 끝날 때까지 다른 작업이 대기**해야 한다.

---

## 동기화(Synchronization)

레이스 컨디션을 방지하고 임계 구역을 올바르게 관리하기 위해 다음 두 가지 조건을 준수해야 한다.

1. **실행 순서 제어**  
   - 프로세스 및 스레드를 올바른 순서로 실행 (예: 레이스 컨디션 방지)
   
2. **상호 배제(Mutual Exclusion)**  
   - 동시에 접근해서는 안 되는 자원에 단 하나의 프로세스/스레드만 접근하도록 보장

---

## 동기화 기법

### 1. 뮤텍스 락(Mutex Lock)
- **상호 배제**를 보장하는 대표적인 동기화 도구
- 임계 구역에 들어가려면 **lock을 획득(acquire)** 해야 하며, 작업이 끝나면 **lock을 해제(release)** 해야 한다.

**작동 방식**
1. 임계 구역 진입 전 `lock.acquire()` 호출  
   - 이미 다른 프로세스/스레드가 lock을 획득했다면, 해제될 때까지 대기  
2. 임계 구역 작업 완료 후 `lock.release()` 호출  

---

### 2. 세마포(Semaphore)
- 공유 자원이 **여러 개** 있을 때 사용하는 동기화 도구
- 신호등처럼 동작하여 **진입 가능 여부**를 제어한다.

**변수**
- `S`: 사용 가능한 공유 자원의 개수  

**wait()**
1. `S` 값을 1 감소  
2. `S > 0` → 임계 구역 진입 가능  
3. `S ≤ 0` → 대기 상태로 전환  

**signal()**
1. `S` 값을 1 증가  
2. `S ≤ 0`이면 대기 중인 프로세스/스레드 중 하나를 깨움  

---

### 3. 조건 변수(Condition Variable)
- **실행 순서 제어**를 위한 동기화 도구  
- 특정 조건이 충족될 때만 프로세스/스레드 실행 가능  

**함수**
- `wait()` : 호출한 프로세스/스레드를 대기 상태로 전환  
- `signal()` : 대기 상태의 프로세스/스레드 실행 재개  

---

### 4. 모니터(Monitor)
- **공유 자원 + 공유 자원 연산(함수)** 으로 구성된 동기화 도구  
- **상호 배제**와 **실행 순서 제어** 모두 가능  

**특징**
- 프로세스/스레드는 반드시 **모니터 내부의 연산을 통해서만 공유 자원 접근** 가능  
- 모니터 내부에서는 동시에 하나의 프로세스/스레드만 실행 가능  
- 이미 다른 프로세스/스레드가 실행 중이라면 큐에서 대기
